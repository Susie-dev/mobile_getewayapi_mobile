## 🌟 一、 简历上该怎么写？ (STAR 法则)

**项目名称：基于 C++11 Reactor 的高性能 API 网关**

**项目描述：** 
该项目是一个轻量级的微服务 API 网关，用于接管前端所有流量。向下游客户端提供 HTTP 协议接入，向上游后端集群提供异步非阻塞的反向代理与负载均衡，同时实现了多种高可用防护插件。

**核心职责与亮点 (技术难点)：**
*   **网络底层**：基于 `epoll` 的边缘触发 (ET) 模式手写了 **One Loop Per Thread** 架构，利用 `eventfd` 实现跨线程异步唤醒，利用 `timerfd` 实现定时器。单机可支撑数万并发长连接。
*   **非阻塞反向代理**：为了解决网关转发时的阻塞问题，从零设计了全异步状态机的 `TcpClient`。在 `connect()` 和 `send()` 阶段将 fd 挂载到 epoll 树上，实现了数据流在客户端和后端服务器之间的零阻塞透传。
*   **智能路由与负载均衡**：设计了 O(1) 时间复杂度的 URL 前缀匹配路由表，并基于 `std::atomic` 实现了无锁的 Round-Robin (轮询) 负载均衡算法，保证高并发下后端流量分发的绝对均匀。
*   **高可用防护插件化**：基于“责任链模式 (Filter Chain)”设计了请求拦截框架。手写了基于**令牌桶 (Token Bucket) 算法**的限流器，有效抵御了瞬时高并发流量冲击；并实现了统一的 HTTP Header JWT 鉴权拦截。

## 🔥 二、 面试官必问的 5 个深水区问题

### Q1: 你的网关是怎么做反向代理的？如果后端服务处理得很慢（比如查数据库耗时 1 秒），你的网关会被卡死吗？
**[回答套路：强调全异步非阻塞]**
> “绝对不会卡死。
> 这是我的项目最大的亮点。普通的做法是收到请求后，用阻塞的 socket 去 connect 后端，这会直接吃掉一个线程。
> 我的做法是：网关在解析完 HTTP 请求后，会创建一个**非阻塞的 `TcpClient`**，调用 `connect()` 后立即返回，并把这个 client_fd 挂载到当前的 `epoll` 上监听可写事件（EPOLLOUT）。
> 线程立刻去处理下一个客户端的请求了。等后端真正连上（或者数据返回）时，epoll 会回调我的 `OnUpstreamMessage` 函数，我再把数据塞到 Downstream Connection 的发送缓冲区里。整个过程是全异步事件驱动的，即使后端卡住，网关的吞吐量也丝毫不受影响。”

### Q2: 你说你实现了“令牌桶限流算法”，如果在多线程环境下，怎么保证高并发请求时不发生“超卖”（令牌数变成负数）？
**[回答套路：解释算法原理 + 并发控制]**
> “我的令牌桶包含两个核心变量：`当前令牌数 tokens_` 和 `上次刷新时间 last_time_`。
> 为了防止超卖，我使用了 `std::mutex` 互斥锁（或者你可以优化为自旋锁/原子操作）。
> 当一个请求到达 `RateLimitFilter::doFilter` 时，我会：
> 1. 加锁。
> 2. 获取当前时间 `now`，计算与 `last_time_` 的差值。
> 3. 差值乘以设定的速率（如每秒 1000 个），算出**这段时间应该新产生多少令牌**，加到桶里（不超过容量上限）。
> 4. 然后判断当前桶里有没有令牌，有就减一放行，没有就直接返回 HTTP 429 拦截。
> 5. 解锁。
> 这种做法的好处是，**我不需要启动一个单独的定时器线程去疯狂加令牌**，而是采用‘惰性计算 (Lazy Evaluation)’的思想，只有在请求真正到来时，才去根据时间差回溯补充令牌，极大地节省了 CPU 资源。”

### Q3: 你的负载均衡算法是怎么实现的？在高并发下加锁了吗？
**[回答套路：原子变量的无锁编程]**
> “我目前实现的是 Round-Robin 轮询算法。
> 它的核心是需要一个索引变量来记录当前该分配给哪台机器。为了避免高并发下多线程争抢这个索引导致性能下降，我**没有使用互斥锁**，而是使用了 C++11 的 `std::atomic<size_t> current_index_`。
> 每次请求到来时，我只需执行 `size_t idx = current_index_++`，这个原子加一操作是硬件级别支持的，性能极高。然后用 `idx % nodes.size()` 取模，就能绝对公平且无锁地找到下一个后端节点。”

### Q4: 你的 Buffer 是怎么解决 TCP 粘包和半包问题的？
**[回答套路：应用层状态机拆包]**
> “因为 TCP 是面向字节流的协议，没有消息边界。
> 对于网关来说，我必须把完整的 HTTP 报文解析出来才能做路由。我在 `Buffer` 类中实现了一个拆包逻辑：
> 由于 HTTP 协议的头部是用 `\r\n\r\n` 作为结束标志的。当 `epoll` 触发读事件时，我把内核缓冲区的数据读到应用层的 `std::string buf_` 中。
> 然后我使用 `buf_.find("\r\n\r\n")` 去寻找边界。如果找不到，说明是个半包，我就直接 return，等下次 epoll 继续读；如果找到了，我就把这部分数据 `substr` 出来作为一个完整的 HTTP Request 对象，并从 Buffer 中 `erase` 掉这部分，从而完美解决粘包。”

### Q5: 如果我要让你的网关支持 HTTPS，你需要改动哪些地方？
**[回答套路：展现你对安全协议的理解]**
> “支持 HTTPS 意味着要在 TCP 层和 HTTP 层之间插入一层 TLS/SSL 加密层。
> 我需要在项目中引入 `OpenSSL` 库。
> 具体改动在 `Connection` 类中：
> 1. 在 `accept` 建立连接后，不能直接读数据，而是要先调用 `SSL_accept()` 完成非阻塞的 TLS 握手。
> 2. 读写数据时，不能用系统的 `read/send`，必须替换为 `SSL_read()` 和 `SSL_write()`，并在发生 `SSL_ERROR_WANT_READ` 时正确地重新注册 epoll 事件。
> 但在实际工业界，为了性能，我们通常会让专门的 Nginx 在最外层做 HTTPS 证书卸载 (SSL Termination)，把剥离后的纯 HTTP 流量打给我们的 API 网关，这样网关可以专心处理业务路由。”

## 🚀 四、 面试真题题库 (50道高频连环追问)

为了让你在面试中无懈可击，我将针对该项目可能涉及的底层网络、并发编程、架构设计等方向，整理了 50 道硬核面试题。建议在面试前逐一过关。

### 第一部分：Reactor 网络模型与 epoll 底层 (1-15题)
1.  **基础**：简述 Reactor 模型的核心思想，与 Proactor 模型有什么区别？
2.  **基础**：你的项目采用的是单 Reactor 还是主从 Reactor 模型？为什么这么设计？
3.  **底层**：`epoll` 和 `select`/`poll` 的本质区别是什么？为什么 `epoll` 性能更高？
4.  **底层**：你的 `epoll` 用的是 LT（水平触发）还是 ET（边缘触发）？两者有什么区别？
5.  **细节**：在 ET 模式下，为什么必须使用非阻塞 (non-blocking) 的 Socket？
6.  **细节**：在 ET 模式下，读取数据时为什么要用 `while` 循环一直读到返回 `EAGAIN` 或 `EWOULDBLOCK`？如果不这么做会怎样？
7.  **架构**：`EventLoop`、`Channel`、`Epoll` 这三个类在你的框架中分别扮演什么角色？它们是如何协作的？
8.  **并发**：你的 `EventLoop` 是如何保证“One Loop Per Thread”的？（提示：判断当前线程 ID）
9.  **并发**：如果工作线程想要向某个 `Connection` 发送数据，但该连接属于另一个 IO 线程，你是怎么处理的？（提示：`queueinloop` 和 `eventfd` 唤醒）
10. **底层**：为什么要用 `eventfd` 来唤醒 EventLoop，而不是用传统的 `pipe` 或者条件变量？
11. **定时器**：你的框架是如何处理超时连接的？（提示：利用 `timerfd`）
12. **底层**：`timerfd` 相比于传统的 `alarm` 信号或者 `sleep` 有什么优势？
13. **异常**：如果客户端突然断网（拔网线），服务器的 `epoll` 能立刻感知到吗？你是怎么处理这种死连接的？（提示：TCP KeepAlive 或应用层心跳）
14. **异常**：什么是 `TIME_WAIT` 和 `CLOSE_WAIT` 状态？你的网关如果出现大量 `CLOSE_WAIT` 通常是什么原因导致的？
15. **性能**：如果你的服务器遭到了 SYN Flood 攻击，底层的 backlog 队列满了，你是怎么处理的？

### 第二部分：HTTP 解析与 Buffer 内存管理 (16-25题)
16. **内存**：介绍一下你设计的 `Buffer` 类的内部结构？它是如何动态扩容的？
17. **内存**：你的 Buffer 在读取数据时，是怎么尽量减少系统调用 `read` 次数的？（提示：`readv` 分散读，结合栈上空间）
18. **协议**：TCP 粘包和半包是怎么产生的？你的框架是如何解决这个问题的？
19. **解析**：在解析 HTTP 请求时，你是如何定位 Header 和 Body 边界的？（提示：寻找 `\r\n\r\n`）
20. **解析**：如果客户端恶意发送一个非常长的 HTTP Header，一直不发 `\r\n\r\n`，你的服务器内存会被撑爆吗？怎么防御？
21. **协议**：HTTP/1.0 和 HTTP/1.1 的 Keep-Alive 机制有什么区别？你的网关支持长连接吗？
22. **性能**：你的网关在转发大文件时（比如 1GB 的视频），是先把 1GB 读到内存再转发，还是流式转发？
23. **内存**：`std::string` 在频繁拼接字符串时会发生内存重分配，你是如何优化的？
24. **智能指针**：你的 `Connection` 对象的生命周期是如何管理的？为什么要用 `std::shared_ptr`？
25. **智能指针**：在 `Connection` 中绑定回调函数时，为什么要继承 `std::enable_shared_from_this` 并使用 `shared_from_this()`？直接传 `this` 指针会发生什么？

### 第三部分：API 网关转发与并发控制 (26-35题)
26. **架构**：详细描述一下从客户端发送请求到网关，再到后端服务，最后返回给客户端的完整数据流转过程。
27. **非阻塞代理**：你的 `TcpClient` 在执行 `connect` 时是非阻塞的，如何判断它到底有没有连接成功？（提示：监听可写事件，并用 `getsockopt` 检查 `SO_ERROR`）
28. **非阻塞代理**：如果后端的微服务挂了（连接被拒绝），你的网关是怎么处理并返回 502 Bad Gateway 的？
29. **连接池**：你的网关每次转发请求都会新建一个到后端的 TCP 连接吗？在高并发下这有什么问题？（提示：短连接耗尽端口，应引入 Upstream 连接池）
30. **线程池**：你的框架中有一个业务线程池，它主要用来做什么？网关的转发逻辑是放在 IO 线程还是业务线程中执行的？
31. **负载均衡**：你实现了哪几种负载均衡算法？简述轮询（Round-Robin）的实现原理。
32. **并发控制**：在实现轮询负载均衡时，你是如何保证无锁线程安全的？（提示：`std::atomic`）
33. **负载均衡扩展**：如果要实现一致性哈希（Consistent Hashing）负载均衡，你应该怎么设计数据结构？（提示：虚拟节点 + 环形哈希空间 + `std::map`的 `upper_bound`）
34. **服务发现**：目前你的后端节点是硬编码在代码里的，如果后端节点动态扩缩容，网关该如何实时感知？（提示：引入配置中心或 Consul/Nacos）
35. **高可用**：如果轮询到的某个后端节点突然宕机了，你的网关会一直把请求发给它吗？如何实现被动健康检查和熔断踢出？

### 第四部分：限流、鉴权与网关高阶特性 (36-45题)
36. **架构**：什么是责任链模式（Chain of Responsibility）？你在项目中哪里用到了它？有什么好处？
37. **限流算法**：除了你使用的令牌桶（Token Bucket）算法，还有哪些常见的限流算法？（提示：漏桶、固定窗口、滑动窗口）
38. **限流对比**：令牌桶算法和漏桶算法最大的区别是什么？应对突发流量时哪个更好？
39. **并发限流**：详细推演一下你的令牌桶限流器在 10 个线程同时竞争令牌时的加锁和时间回溯过程。
40. **分布式限流**：如果你的 API 网关部署了 5 台机器，怎么实现全局限流（比如总共只允许 1000 QPS）？（提示：基于 Redis + Lua 脚本）
41. **鉴权**：你的 `AuthFilter` 是如何拦截非法请求的？
42. **安全**：什么是 JWT（JSON Web Token）？为什么现代微服务网关倾向于用 JWT 而不是传统的 Session/Cookie？
43. **协议转换**：如果前端是 HTTP 请求，后端是 gRPC 服务，网关层面需要做什么工作？
44. **熔断降级**：什么是服务熔断？你的网关有实现类似 Hystrix 的熔断机制吗？思路是什么？
45. **日志**：高并发网关写日志如果用普通的 `printf` 或 `fprintf` 会有什么致命问题？异步日志库（如 spdlog）的核心原理是什么？

### 第五部分：项目实战与系统调优 (46-50题)
46. **压测**：你用什么工具对网关进行了压测？压测时的系统拓扑是怎样的？
47. **性能瓶颈**：在压测过程中，你发现系统的瓶颈在哪？是 CPU、内存带宽还是网络 I/O？你是如何定位的？（提示：使用 `top`, `perf`, `strace`, 火焰图）
48. **调优**：为了达到更高的并发，你对 Linux 内核参数做过哪些调优？（提示：`ulimit -n` 修改文件描述符上限，`tcp_tw_reuse`，`somaxconn` 等）
49. **架构演进**：如果现在的单机网关抗不住了，整个系统架构应该如何横向扩展？（提示：在网关前面再加一层 L4 负载均衡，如 LVS 或 F5）
